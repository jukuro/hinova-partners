# Hinova Partners パートナーポータル化 計画

作成日：2026年6月25日

## 背景・ゴール

現状は**管理者専用アプリ**（メール+パスワードでログイン、全データに管理者がアクセス）。
これを拡張し、**パートナー本人がログインして使えるポータル**を追加する。

実現したいこと：
1. 紹介フォームで**商材を複数選択**できる
2. パートナーを**電話番号＋名前だけで簡単登録 → SMSで招待URL → 本人が本登録**
3. 本登録時に**ホーム画面に追加（PWA）**を促す
4. パートナーが**ユーザーページ（ポータル）**で、自分の紹介管理・ランク確認・商材紹介・資料DLができる
5. 管理者は**管理画面**で全体売上・紹介状況・全パートナーのステータスを管理できる

## 確定した方針

| 項目 | 決定 |
|---|---|
| SMS送信 | `sms:`リンク（管理者のスマホでSMSアプリが定型文+URL付きで開く。無料・手動） |
| パートナー認証 | 招待トークンURL → 本登録でメール+パスワード設定。以降はメール+パスワードでログイン |
| 資料アクセス | パートナーのランク・担当商材で出し分け |

## 1. 認証・権限の設計

### 現状
- `partner_admins` テーブル＋ `is_partner_admin()` 関数で管理者判定
- ログインユーザー＝全員管理者前提

### 変更
- `partners` テーブルに `auth_user_id uuid`（Supabase authユーザーとの紐付け）と `invite_token text`、`invited_at`、`activated_at` を追加
- ログインユーザーの種別を判定：
  - `is_partner_admin()` が true → **管理者**（全データアクセス、管理画面へ）
  - `partners.auth_user_id = auth.uid()` に一致 → **パートナー**（自分のデータのみ、ポータルへ）
- RLS を**自分のデータのみ**に絞るポリシーへ拡張（partners/deals/leads/commissions/partner_products/partner_commission_rules を `auth_user_id` 基準で）

### ルーティング
- ログイン後、種別で振り分け：管理者 → `/dashboard`、パートナー → `/portal`
- パートナーは管理画面ルートにアクセス不可（ガード）

---

## 2. パートナー簡単登録 → SMS招待 → 本登録

### 2-1. 管理者側：簡単登録＋SMS招待（Partners.jsx 拡張）
- パートナー新規作成を**「名前＋電話番号だけ」**でも保存可能に（既存フォームの必須を緩和）
- 保存時に `invite_token`（`crypto.randomUUID()`）を発行、`status='reviewing'`
- 一覧の各行に**「SMSで招待」ボタン**：
  - 招待URL `${origin}/portal/setup/{invite_token}` を生成
  - `sms:{電話番号}?&body={定型文+URL}` を開く
  - 電話番号未登録なら通知して中断
- 「URLをコピー」ボタンも併設（SMS以外で送りたい場合）

### 2-2. パートナー側：本登録ページ（新規 PartnerSetup.jsx）
- ルート `/portal/setup/:token`（未ログインでアクセス可）
- トークンから対象 `partners` 行を引き、本人確認（電話番号の下4桁入力などの軽い確認は任意）
- 入力：**メールアドレス＋パスワード**（Supabase auth ユーザー作成）＋基本情報（フリガナ・会社/屋号・住所・振込先など）
- 成功時：`partners.auth_user_id` を作成した auth ユーザーに紐付け、`activated_at` 設定、`status` は `reviewing` のまま（管理者が稼働中へ）
- 完了後、**PWA「ホーム画面に追加」案内**を表示（InstallPrompt）

### 2-3. パートナーログイン（Login.jsx 拡張 or 共通）
- メール＋パスワードでログイン（既存の仕組みを流用）
- ログイン後、パートナーはポータルへ

## 3. PWA 対応（ホーム画面に追加）

現状 Partners アプリに manifest/Service Worker なし。Hinova Biz の構成を移植：
- `public/manifest.webmanifest`（name: Hinova Partners、アイコン）
- `public/sw.js`（最小キャッシュ）＋ 登録スクリプト
- `index.html` に manifest / theme-color / apple-mobile-web-app メタ
- アイコン生成（既存 `scripts/gen-icons.mjs` 相当があれば流用、なければ簡易生成）
- `InstallPrompt.jsx`（Hinova Biz から移植）を本登録完了画面とポータルに表示

## 4. パートナーポータル（新規・スマホ前提）

レイアウト：`PortalLayout.jsx`（ボトムナビ or シンプルなヘッダ）。
ページ構成：

| ページ | 内容 |
|---|---|
| ホーム `/portal` | 自分のランク・今月の紹介数・お礼予定額・クイック紹介ボタン |
| 紹介する `/portal/refer` | 商材を選んで（複数可）紹介先の名前・連絡先を入力 → 送信。紹介URL/QRの表示も |
| 紹介状況 `/portal/referrals` | 自分が紹介した人の一覧とステータス（受付/対応中/利用開始など） |
| お礼 `/portal/rewards` | 自分のお礼履歴（対象月・商材・お礼額・支払状況） |
| 資料 `/portal/materials` | 自分のランク・担当商材で閲覧可能な説明資料のDL |
| マイページ `/portal/profile` | 基本情報の確認・編集 |

- 表現は「営業」ではなく「紹介」、「ノルマ」表現は使わない（仕様書 第17節準拠）
- ランクは表示するが、加算率の細かい数字より「今のランク」を分かりやすく

## 5. 資料アクセスの出し分け

- `materials` に `min_rank_sort int`（このランク以上で閲覧可、null=全員）と `business`（商材事業に紐付け）を追加（`business` は既存）
- パートナーポータルの資料一覧は：
  - `is_public = true` かつ
  - （`min_rank_sort` が null または パートナーのランク sort_order 以上）かつ
  - （`business` が null または パートナーの担当商材の事業に含まれる）
- 管理画面の資料管理（Materials.jsx）に `min_rank_sort` 設定UIを追加

---

## 6. 紹介フォームの複数商材選択

- 管理者の「紹介状況」「かんたん紹介」、およびパートナーポータルの「紹介する」で**商材を複数選択**（チェックボックス）
- 保存時は**商材ごとに1件の紹介レコードを作成**（fan-out）
  - 理由：報酬は商材ごとに率・お礼額が異なり、契約日ロック・履歴も商材単位のため、1商材=1紹介ラインが最もクリーン
- 既存の `deals.product_id` / `leads.product_id`（単一）構造を変えずに済む

---

## 7. 管理画面の強化

ほぼ現状で要件を満たすが、以下を追加：
- ダッシュボードに**パートナー別ステータス一覧**（稼働中/審査中/停止、今月紹介数、累計お礼額）
- 全体売上（＝決済金額合計）と紹介状況サマリーの表示強化
- パートナー一覧で**招待状況**（未招待/招待済み/本登録済み）が分かるバッジ

## 8. DBマイグレーション（新規 20260626_partner_portal.sql）

```sql
-- partners: 認証紐付け・招待
alter table public.partners
  add column if not exists auth_user_id uuid,
  add column if not exists invite_token text unique,
  add column if not exists invited_at timestamptz,
  add column if not exists activated_at timestamptz,
  add column if not exists address text,
  add column if not exists bank_info jsonb;  -- 既存なら不要

-- materials: ランク出し分け
alter table public.materials
  add column if not exists min_rank_sort int;

-- パートナー本人判定
create or replace function public.current_partner_id()
returns uuid language sql stable security definer set search_path = public as $$
  select id from public.partners where auth_user_id = auth.uid() limit 1;
$$;

-- RLS：管理者は全件、パートナーは自分の行のみ（partners/deals/leads/commissions/...）
-- 例（partners）：
--   using ( is_partner_admin() or auth_user_id = auth.uid() )
-- deals/leads/commissions は partner_id = current_partner_id() を許可
```

## 9. フェーズ分割と進め方

| Phase | 内容 | 目安 |
|---|---|---|
| **P1** | 複数商材選択（管理者の紹介フォーム fan-out）※すぐ効果が出る | 1h |
| **P2** | DBマイグレーション（auth紐付け・招待・RLS・current_partner_id） | 1.5h |
| **P3** | 管理者：簡単登録＋SMS招待ボタン＋URLコピー | 1.5h |
| **P4** | パートナー本登録ページ（トークン→メール+PW作成→紐付け） | 2.5h |
| **P5** | PWA対応（manifest/sw/InstallPrompt/アイコン） | 1.5h |
| **P6** | パートナーポータル（レイアウト＋6ページ） | 4h |
| **P7** | 資料の出し分け（min_rank_sort＋ポータル資料DL＋管理UI） | 1.5h |
| **P8** | 管理画面強化（パートナー別ステータス・招待状況バッジ） | 1.5h |
| **P9** | ビルド・実機確認 | 1h |
| 合計 | | 約16h |

おすすめ：**P1 → P2 → P3 → P4 → P5** をまず通し、SMS招待〜本登録〜ホーム画面追加の一連の流れを動く状態にしてから、P6以降のポータル中身を作り込む。

## 10. 事前にお願いしたいこと（ユーザー作業）

- Supabase ダッシュボードで、本登録時の**メール確認（Confirm email）の扱い**を決める
  - 確認必須だと、本登録直後にメール内リンクのクリックが必要になり摩擦が増える
  - 推奨：パートナー本登録は確認不要（または後日確認）に設定
- 管理者メールを `partner_admins` に登録済みであること（既存の仕組み）
